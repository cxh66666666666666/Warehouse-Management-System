# first_drogon_app 项目路线图

## 1. 项目定位

这个项目不建议做成一个“什么都有一点，但每样都不深”的练手项目。

更适合把它打造成一个可以写进简历、并且有技术亮点的项目：

**基于 Drogon + MySQL 的订单与库存后端系统**

核心关键词：

- 商品管理
- 库存管理
- 订单创建
- 事务一致性
- 乐观锁防超卖
- 接口测试
- 日志与监控
- Docker 部署
- 可选 Qt6 管理端

如果后面做得扎实，这个项目在简历上可以往这个方向表达：

> 独立设计并实现基于 C++20 / Drogon / MySQL 的订单库存后端系统，支持商品管理、订单创建、事务处理、库存扣减与乐观锁防超卖，具备测试、日志、监控与容器化部署能力。

- 已有商品相关接口

## 4. 项目主线建议

推荐你把项目主线固定成下面这个方向，不要频繁改题：

**一个库存敏感的下单系统**

场景可以理解为：

- 管理商品信息
- 管理库存信息
- 用户提交订单
- 服务端校验商品和价格
- 扣减库存
- 创建订单主表和订单详情表
- 防止超卖
- 返回订单结果

这个方向适合你当前的代码基础，也适合做出技术亮点。

## 5. 分阶段任务路线

下面的任务按顺序推进。不要跳太快。

### 阶段 0：先把项目基础盘稳住

**目标**

让项目具备“别人拉下来就知道是什么、怎么跑、目录是什么”的基础工程感。

**我要做什么**

- [done] 把 `README.md` 改成真实项目说明，不再使用模板内容
- [done] 画清楚当前目录职责：`controllers`、`services`、`models`、`test`
- [done] 明确数据库表结构，并把建表 SQL 整理成文档或脚本
- [done] 明确当前已有接口及其用途
- [done] 统一接口返回格式
- [done] 统一错误码和错误信息风格
- [done] 把配置项说明清楚，例如端口、数据库连接、日志级别

**我要学什么**

- CMake 基础：目标、源文件、子目录、测试目录怎么组织
- C++ 工程结构：头文件和源文件的职责划分
- HTTP 基础：请求方法、状态码、JSON 返回
- REST API 基础设计
- MySQL 基础：主键、外键、索引、事务

**完成标准**

- 新同学打开仓库后，5 到 10 分钟内知道项目做什么
- 你自己能清楚说出每个目录负责什么
- 你能写出当前核心表结构和字段含义

**这一阶段的简历价值**

还不高，但这是后面所有含金量的地基。

---

### 阶段 1：把商品、库存、订单三个核心模块补完整

**目标**

先完成一个能闭环的业务系统，而不是零散接口集合。

**我要做什么**

- [ ] 商品模块补齐：新增、查询、修改、删除、分页列表
- [ ] 库存模块补齐：查询库存、增加库存、扣减库存、库存校验
- [ ] 订单模块补齐：创建订单、查询订单详情、订单列表
- [ ] 明确 DTO / JSON 字段，避免请求结构混乱
- [ ] 做统一参数校验
- [ ] 明确金额单位，统一使用“分”而不是浮点
- [ ] 给订单接口补充失败场景处理

**我要学什么**

- DTO 和领域对象的区别
- 参数校验思路
- 金额为什么不应该用浮点数
- 数据库表关系设计：商品、库存、订单主表、订单详情表
- 分页查询和索引的基础思路

**完成标准**

- 你可以完整演示一次“查询商品 -> 下单 -> 扣库存 -> 查询订单”
- 接口文档和实际实现是一致的
- 常见异常输入能返回合理错误

**这一阶段的简历价值**

你已经不再只是做 CRUD，而是在做一个小型业务系统。

---

### 阶段 2：把业务一致性做成项目亮点

**目标**

这是项目开始“有含金量”的关键阶段。

**我要做什么**

- [ ] 梳理下单流程里的事务边界
- [ ] 完整处理库存扣减失败时的回滚
- [ ] 明确乐观锁的使用方式和版本字段含义
- [ ] 补充并发下单场景说明
- [ ] 设计幂等思路，避免重复下单
- [ ] 设计订单状态流转，例如待创建、成功、失败、取消
- [ ] 区分业务错误、参数错误、数据库错误

**我要学什么**

- MySQL 事务的 ACID
- 隔离级别的基本概念
- 乐观锁和悲观锁的区别
- 超卖问题的常见解法
- 幂等性的意义
- 业务异常和系统异常的划分

**完成标准**

- 你能讲清楚为什么库存不会被随便扣错
- 你能讲清楚为什么失败后不会留下半成品数据
- 你能讲清楚乐观锁在这个项目里解决了什么问题

**这一阶段的简历价值**

这一阶段非常重要，因为它能把项目从“普通后台管理”拉到“懂业务一致性”的层级。

---

### 阶段 3：建立测试体系

**目标**

让项目从“我自己试过能用”变成“我有证据证明它能用”。

**我要做什么**

- [ ] 为参数校验写单元测试
- [ ] 为订单创建主流程写集成测试
- [ ] 为库存不足、商品不存在、金额不一致写失败测试
- [ ] 为并发扣库存写场景测试
- [ ] 给关键 service 层建立测试样例
- [ ] 约定测试数据初始化方式

**我要学什么**

- 单元测试和集成测试的区别
- 什么是测试金字塔
- 如何设计正常路径和异常路径测试
- 什么是边界值
- 为什么业务系统要重点测失败分支

**完成标准**

- 关键流程至少有一组成功测试
- 关键失败场景都有测试覆盖
- 你改动代码后可以用测试快速回归

**这一阶段的简历价值**

有测试的项目，比没有测试的项目更像正式工程，面试里也更容易加分。

## 8. 推荐的实际推进顺序

如果你想要一个更直接的执行顺序，建议按下面来：

1. 先重写 `README.md`
2. 补齐商品、库存、订单三个模块的接口闭环
3. 整理数据库表结构和索引
4. 把订单创建流程做扎实，尤其是事务和库存扣减
5. 建立测试
6. 补日志、健康检查、基础监控
7. 做 Docker 和部署文档
8. 最后再决定要不要加 Qt6 管理端或 Redis 缓存

---

## 9. 不建议现在立刻做的事

为了避免项目失控，下面这些事不建议太早做：

- 不要一开始就上微服务
- 不要一开始就追求很多中间件
- 不要先做复杂前端界面
- 不要把时间大量花在“炫技”而不是“把主流程做扎实”
- 不要只做 happy path，不测失败分支

你现在最值钱的能力，不是堆技术名词，而是把一个小而完整的后端系统做扎实。
