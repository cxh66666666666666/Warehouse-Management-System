### P1：部分表述有概念性误差

#### 4. `docs/directory_description.txt:3`

这里写到：

> 这样就不需要再在代码中查询数据库，只需要简单调用类函数就可以拿到字段

这句话不准确。

`models/` 里的 ORM 类不是“自动把数据库内容带进来”，而是提供了一个**表结构映射**。真正拿到数据，仍然需要：

- 用 `Mapper`
- 或执行 SQL
- 或调用 ORM 查询接口

也就是说，`models` 负责的是“数据结构映射”，不是“自动查询数据库”。

#### 5. `docs/directory_description.txt:1`

你已经意识到当前 `controllers` 里塞了不少数据库处理代码，这个判断是对的，而且很关键。

不过这里还可以更进一步：

- `controller` 负责 HTTP 层职责
- `service` 负责业务编排
- `model` 负责数据映射

你现在已经在看见“代码能跑”和“分层合理”之间的差距了，这非常好。

#### 6. `docs/configuration_item.md:11` 和 `docs/configuration_item.md:26`

这里同时使用了：

- `threads_num`
- `connection_number`

但当前项目实际配置里用的是：

- `number_of_threads`
- `number_of_connections`

虽然你写了“或”的说明，但如果是项目文档，建议优先写**当前仓库实际在用的配置键名**，避免以后照文档改配置时改错。

## 6. 最需要改进的能力点

从这批文档里，我最建议你接下来重点提升这三种能力：

### 1. 区分“现状”和“目标”

以后写文档时，建议你强制分为两类：

- 当前实现
- 设计中的目标规范

这样文档会立刻清楚很多。

### 2. 区分“技术事实”和“个人理解”

例如：

- “当前配置里实际键名是什么”属于技术事实
- “我认为生产环境推荐这样配置”属于经验建议

这两类内容都能写，但不要混成一层。

### 3. 把口语化理解升级成工程化表述

例如“服务员”“翻译成类”这种说法，有助于你自己理解，但如果放进正式文档，就需要换成更工程化的表达：

- 请求接入层
- 业务编排层
- 数据映射层
- 测试验证层

## 7. 建议你下一轮怎么写

下一轮继续写文档时，建议按下面这个标准来：

### 对“当前实现类文档”

必须满足：

- 和代码一致
- 和配置一致
- 和接口行为一致

### 对“设计方案类文档”

必须满足：

- 明确标注是“设计中”或“目标规范”
- 不写成“已经完成”
- 能解释为什么要这样设计

### 对“学习总结类文档”

必须满足：

- 说清你理解了什么
- 说清你还没理解什么
- 不要冒充事实文档
